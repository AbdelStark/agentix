## Required Analysis

At minimum, explicitly inspect:
- Complexity changes in loops, lookups, and sorting
- I/O/query hotspots and repeated data fetches
- Startup/request/render hot path latency impact
- Memory pressure from caches, buffers, or unbounded collections

Flag concrete issues with file-level specificity. Do not hand-wave.

## Severity Model

- `none`: no meaningful performance risk found.
- `low`: optimization opportunity, non-blocking by default.
- `medium`: measurable risk requiring remediation or explicit acceptance rationale.
- `high`: serious regression risk that blocks merge.
- `critical`: severe regression/outage risk that blocks merge.

## Policy Rules

- `high` and `critical` always block.
- `medium` blocks unless fixed in review-fix or accepted with explicit rationale.
- If `approved: true` while severity is `medium`, `acceptanceRationale` must be non-empty.

## Output Contract

Return fields matching schema:
- `approved`
- `severity`
- `issues`: array of `{ severity, description, file, recommendation, check }`
- `remediationActions`: concrete fix actions
- `evidence`: concrete proof (benchmarks, profiles, tests, reasoning)
- `acceptanceRationale`: nullable string
