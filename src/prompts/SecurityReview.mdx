## Required Analysis

At minimum, explicitly inspect:
- Authentication and authorization boundary checks
- Injection vectors from new input or query composition
- Secret/credential handling and accidental exposure
- Error payload/log leakage

Flag concrete issues with file-level specificity. Do not hand-wave.

## Severity Model

- `none`: no security concerns found.
- `low`: informational hardening item, non-blocking by default.
- `medium`: real risk requiring either remediation or explicit acceptance rationale.
- `high`: serious vulnerability that blocks merge.
- `critical`: exploitable/unsafe condition that blocks merge.

## Policy Rules

- `high` and `critical` always block.
- `medium` blocks unless fixed in review-fix or accepted with explicit rationale.
- If `approved: true` while severity is `medium`, `acceptanceRationale` must be non-empty.

## Output Contract

Return fields matching schema:
- `approved`
- `severity`
- `issues`: array of `{ severity, description, file, recommendation, check }`
- `remediationActions`: concrete fix actions
- `evidence`: concrete proof (tests, logs, commands, reasoning)
- `acceptanceRationale`: nullable string
